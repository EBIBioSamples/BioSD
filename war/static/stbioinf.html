SampleTab format for submitting Bioinfomaticians

Note: Specification of this file format is at an early stage. All details are subject to flux, depending on how useful they are in the face of real data.

In addition to the information provided for biologists, bioinfomaticians should bear in mind the additional information on this page when creating SampleTab files and/or related tools.

There are some general rules that apply to all SampleTab files. These describe overall properties of the file, rather than particular content.:

  • Cells are tab separated and only tab separated.
  • Empty lines are ignored, including lines composed only of whitespace (inc tabs)
  • If the first character on a line is `#`, then that line is ignored. Use of `#` characters elsewhere is allowed.

The following are "best practice" guidelines and should be followed:

  • Where possible MSI and SCD sections should be in a single file.
  • Line endings can be in any format (Windows, Linux, Mac), but should be consistent within a file.
  • Dates and times should be in ISO 8601 format YYYY-MM-DD HH:MM:SS.ss (omitting excessive accuracy when appropriate)
  • Files must be UTF-8 encoded. This allows for accents and other characters.
  • Filenames should end with .sampletab.txt
  • Numeric data should use a full stop as a decimal separator and no other separators. For example, 1234.56 rather than 1234,56 or 1,234.56. Units should be specified in the following column under a Units tag.

Meta Submission Information (MSI) ^

There is some additional MSI tags that can be used. These are aimed at advanced users.

  • Submission Update Date
    • Date when submission was last modified.
    • Not supplied with submissions, but may be seen when accessing.
  • Term Source Name
    • Local name for each term source (ontology or other database)
    • Optional if no additional term sources are used.
  • Term Source URI
    • Must be a valid URI.
    • URI (e.g. web address) at which the Term Source may be accessed.
    • Optional if no additional term sources are used.
  • Term Source Version
    • Version number of the term source used.
    • Must be a valid version number for this term source.
    • If the term source has no version numbering, use the date of access.
    • Optional if no additional term sources are used.
  • Database Name
    • Name of the database(s) data this submission came from, if any
  • Database URI
    • URI of the submitting database(s)'s front pages
  • Database ID
    • Identifier or Accession of this submission within the submitting database
    • May be omitted if the database uses the same Submission Tracking Identifier

Sample Characteristics and Description (SCD) ^

In addition to the notes for biologists, the following additional information may be on interest for more complicated collections of samples.

There are two types of entity in the SCD section - Sample and Group. Each entity is a single row starting from a column Sample Name or Group Name respectively.

Samples contain all data in cells to the right until another Sample Name column or the end of the spreadsheet These columns can include one or more Group(s) that the sample is a "member of". If a sample is on the same row as another sample, it is assumed that the sample to the right has a "derived from" relationship with the sample to the left. This can be repeated to chain several samples together, for example a human subject and a blood sample. Samples can be pooled and/or split by duplicating the same sample data (including name) over multiple rows.

Groups contain all data in cells to the right until a Sample Name or Group Name column. Groups do not have to contain Samples e.g. if data is confidential. Groups should only be used when they contain information not specified elsewhere. I.e. samples should not be grouped by SCD tag data (e.g. organism, tissue, disease state, etc). One example is where there is no information available for individual samples.

Both Samples and Groups can have Tags. Tags are specified as columns to the right of Sample Name or Group Name. Some recommended sets of tags will be developed for certain types of data.

  • Group Name
    • Group within this submission that the sample belongs to.
    • If multiple rows have the same Group Name, all other information about that group must be identical.
    • If a sample belongs to multiple groups, use multiple Group Name columns.
  • Group Description
    • Optional
    • Free text description of this group.
    • Should not include information that can be specified elsewhere, e.g. age, project, disease state, ethnic group, etc.
  • Group Size
    • If the group exists, but no specific sample information is available e.g. due to privacy or lack or records.
    • In such cases, the group should have appropriate attributes and can use the value "private" to indicate it is known but not available.
    • Must be an integer > 0 or "private".

Tags can be labelled as as a comment or a characteristic for easier classification. Comment[x] (e.g. Comment[Data quality]) should be used to label attributes that do not directly refer to the sample or should not be used to categorize samples, but are recorded for future reference. Characteristics (e.g. Characteristic[Phenotype]) can be used to label attributes that can be used to categorize samples - e.g. Characteristic[Organism], Characteristic[Sex], etc. If not labelled, a tag is a characteristic by default.

Attributes and attribute data can be linked to Ontology entries. In such cases, the columns containing the attribute that is an ontology term should be Term Source REF and should contain a name from the Term Source Name section of the MSI. The value in the data that is linked to the ontology should be an ontology term from the specified ontology, either as an identifier or the term as text. Terms can be looked up in multiple ontologies simultaneously through OLS.
